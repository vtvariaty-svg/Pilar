Objetivo:
Corrigir a separação entre usuário administrador e usuário comum/cliente.

Regra de negócio obrigatória:
- O e-mail vtvariaty@gmail.com é o único super_admin inicial da plataforma.
- Apenas o super_admin pode criar/promover outros usuários admin.
- Usuários comuns/clientes não podem acessar nenhuma rota /admin.
- Usuários comuns/clientes só poderão acessar futuramente a área /cliente.
- Usuário autenticado não significa admin.
- Admin precisa existir em Firestore em adminUsers/{uid} com active true.

Auditoria já identificou:
1. ProtectedRoute atual só verifica se existe usuário autenticado.
2. AdminLogin redireciona qualquer usuário autenticado para /admin.
3. Firestore Rules já têm adminUsers, mas o front-end não consulta adminUsers.
4. Storage ainda permite read/write para qualquer autenticado fora de /portfolio.
5. npm run lint existe, mas eslint não está em devDependencies.
6. Botão de agendamento da calculadora navega para /#contato, mas a seção atual usa id="orcamento".

7. Corrigir Firestore Rules

Atualizar rules com:
- signedIn()
- isBootstrapSuperAdmin()
- isAdmin()
- isSuperAdmin()

Regras:
- adminUsers:
  - super_admin lê todos
  - usuário lê próprio adminUser
  - bootstrap permite vtvariaty@gmail.com criar o próprio super_admin
  - apenas super_admin cria/atualiza/desativa admins

- users:
  - usuário cria/lê próprio perfil
  - admin lê usuários
  - super_admin gerencia usuários

- leads, quoteEstimates, appointments:
  - create público controlado
  - read/update/delete apenas admin
  - futuramente cliente poderá ler documentos vinculados a customerUid

- pricingSettings, serviceCatalog, platformSettings:
  - read público quando necessário
  - write apenas admin
